## Architecture Decisions

### ADR-000007 — Multi-Repository Architecture Adopted
**Decision:** LES is structured across multiple repositories — one per module plus operational, infrastructure, and master repositories.
**Rationale:** Enterprise systems separate concerns. A monorepo at this scale would be unmanageable. Each repository has its own version history, access control, and documentation.
**GitBook:** Yes

### ADR-000008 — Security is a Cross-Cutting Concern
**Decision:** Security is embedded in every repository from Phase 0. LES-SEC manages governance, risk, and compliance as a functional module.
**Rationale:** Security retrofitted after the fact is always inadequate. Treating it as a cross-cutting concern from day one reflects real enterprise architecture practice.
**GitBook:** Yes

### ADR-000009 — LES-INFRA, LES-API, LES-PORT Separated from Modules
**Decision:** Infrastructure, API gateway, and portfolio website are operational repositories — not modules.
**Rationale:** These components serve all modules. They belong to no single module. Separating them prevents false dependency relationships and reflects their actual architectural role.
**GitBook:** Yes

### ADR-000017 — Full Document Taxonomy Coding Adopted — Option B
**Decision:** Industry-standard notation with type-prefixed six-digit codes. ADR prefix for decision records. Module/function code plus numeric suffix for all other document types.
**Rationale:** Scales to 999,999 documents per type. Document type immediately identifiable from code. Signals professional familiarity with enterprise documentation practices.
**GitBook:** Yes

### ADR-000020 — 18 Repositories Confirmed — Current Count
**Decision:** 18 repositories confirmed as the Phase 0 baseline: 10 modules, 4 operational, LES-Master, LES-DASH, LES-AGT, LES-demo-data. Plus NikiDigitals profile README.
**Rationale:** Repository count evolved across multiple sessions as architecture was refined. This records the confirmed Phase 0 baseline. New repositories may be added in future phases.
**GitBook:** No

### ADR-000031 — GitHub and GitBook Both Contain Documentation — Different Audiences
**Decision:** GitHub holds working documentation for technical audiences. GitBook holds published documentation for architects, researchers, and serious readers. Same subject, different treatment. Neither replaces the other.
**Rationale:** Each platform attracts readers with different expectations. Writing for both audiences equally from a single document produces content that serves neither well.
**GitBook:** Yes

### ADR-000033 — GitHub Universal Folder Structure Confirmed
**Decision:** Every repository follows the universal structure: docs/ with architecture/, decisions/, governance/, research/, roadmap/, specs/ subfolders. src/ with tests/. journal/ at root. One TEMPLATE.md per folder.
**Rationale:** Predictable structure across 18 repositories. Any developer navigating repositories always knows where to find documentation of a given type.
**GitBook:** No

### ADR-000034 — LES-AGT is a Custom-Built Multi-Agent System
**Decision:** LES-AGT is built from first principles. Azure OpenAI Service provides LLM infrastructure only. Agent architecture, orchestration, memory, and decision logic are designed and built as part of LES.
**Rationale:** The distinction between using an agent framework and building one is fundamental to the portfolio value. Semantic Kernel and AutoGen are studied for patterns — not adopted.
**GitBook:** Yes

### ADR-000036 — GitBook Security & GRC and AI Agents as Dedicated Top-Level Sections
**Decision:** Security & GRC and AI Agents are elevated to dedicated top-level GitBook sections. New top-level sections require an ADR before creation.
**Rationale:** Both are cross-cutting concerns that span all modules. Neither belongs inside a single module section. Their elevation reflects their architectural significance.
**GitBook:** No

### ADR-000040 — GitBook Structure Finalised at 12 Top-Level Sections
**Decision:** GitBook space structured with 12 top-level sections. Section 12 is Articles & Publications.
**Rationale:** 12 sections covers all current and anticipated content types for the five-year project horizon. Controlled structure growth — new sections require an ADR.
**GitBook:** No

### ADR-000044 — Demo Environment on Separate Azure Subscription — Phase 2
**Decision:** Fully isolated demo environment on a separate Azure subscription. Synthetic persona for all public demonstration. Personal data never exposed.
**Rationale:** Resolves the tension between building in public and processing personal data. Subscription-level isolation is the strongest available boundary in Azure.
**GitBook:** Yes

### ADR-000047 — Public Identity Architecture Formalised
**Decision:** NikiDigitals is the permanent professional identity. LES is the flagship project beneath it. Each portfolio layer speaks at the correct level of this hierarchy.
**Rationale:** Without a clear hierarchy the relationship between NikiDigitals and LES is ambiguous. The two-level architecture resolves this permanently.
**GitBook:** Yes

### ADR-000051 — LES-DASH Established as a Dedicated Repository
**Decision:** LES-DASH is a dedicated repository separate from LES-PORT. Hybrid approach: Power BI in Phase 1–2, custom React dashboard from Phase 3.
**Rationale:** The dashboard is a complex application in its own right — its own codebase, deployment pipeline, and documentation lifecycle.
**GitBook:** Yes

### ADR-000061 — LES-KNW Identified as Future Native Documentation Database
**Decision:** LES-KNW absorbs the documentation function from Phase 3. GitHub and GitBook serve Phase 0–2. Current documentation structure mirrors the eventual LES-KNW schema.
**Rationale:** The knowledge management module is the natural permanent home for LES documentation. Designing current structure to mirror the schema makes Phase 3 migration an import, not a restructure.
**GitBook:** Yes

### ADR-000062 — Diagramming Standards Adopted
**Decision:** C4 Model for system architecture, BPMN 2.0 for process flows, ERD for data models, Azure Architecture icons for infrastructure. draw.io as primary tool.
**Rationale:** No single notation covers all diagram types required by a production-grade ERP. Each standard is the recognised professional convention for its domain.
**GitBook:** Yes

## Strategic Decisions

### ADR-000004 — Four-Layer Portfolio Architecture
**Decision:** Portfolio structured across four layers — Website (executive), GitBook (thinking), GitHub (evidence), LinkedIn (professional).
**Rationale:** Each layer serves a distinct audience and purpose. Together they route any visitor to the appropriate depth of content for their intent.
**GitBook:** Yes

### ADR-000010 — NikiDigitals Covers All Projects
**Decision:** NikiDigitals is the professional brand covering Finance, Systems, and Technology. It is broader than LES and will contain multiple projects over time.
**Rationale:** Building the brand around a single project creates a naming problem when future projects are added. A broader brand identity scales cleanly.
**GitBook:** No

### ADR-000014 — LES is a Portfolio Flagship — Architecture Preserves Scaling Optionality
**Decision:** LES is built to production-grade standards for portfolio purposes. Productisation is not a current objective. Architecture preserves that option if the decision changes.
**Rationale:** Conflating portfolio development with product development creates pressure that distorts architectural decisions. Keeping them separate produces better outcomes for both.
**GitBook:** Yes

### ADR-000015 — Career Target Confirmed — Finance Transformation Architect
**Decision:** Finance Transformation Architect, specialising in complex enterprise finance systems at scale.
**Rationale:** Every architectural, learning, and documentation decision is evaluated against this target. If it does not make a better Finance Transformation Architect, it is deferred or dropped.
**GitBook:** Yes

### ADR-000043 — Building in the Open — Confirmed as Deliberate Strategy
**Decision:** LES is built publicly. Every commit, decision, and journal entry is publicly accessible. The beginner starting point is named explicitly.
**Rationale:** A five-year public build record creates timestamped, auditable capability evidence that cannot be faked. Naming the starting point is a strength, not a weakness.
**GitBook:** Yes

### ADR-000048 — LinkedIn Personal Profile as Primary Portfolio Destination
**Decision:** Personal profile (linkedin.com/in/nmlangendam) is primary. Company page (linkedin.com/company/niki-digitals) is secondary.
**Rationale:** Professional audiences connect with people, not company pages. The personal profile carries the professional narrative.
**GitBook:** No

## Operational Decisions

### ADR-000003 — Certifications Only Where Genuinely Valuable
**Decision:** Certifications pursued only where they add genuine value to knowledge, credibility, or the portfolio. Not collected as vanity metrics.
**Rationale:** Quality over quantity. The certification pathway is deliberately focused on what serves the Finance Transformation Architect target.

### ADR-000005 — UK English for All Outputs
**Decision:** All LES documentation, website content, and professional communications use UK English.
**Rationale:** Professional consistency across all published work.

### ADR-000006 — No Shortcuts on Understanding
**Decision:** Every layer is understood before it is built. Depth of understanding is the standard, not speed of delivery.
**Rationale:** Systems built on shallow understanding require rebuilding. The five-year timeline exists precisely to develop genuine depth.

### ADR-000011 — Insights Replaces Blog
**Decision:** The website content section is titled Insights, not Blog.
**Rationale:** Executive-level tone. Blog undersells the content.

### ADR-000012 — All Repositories Under NikiDigitals Personal Account
**Decision:** No separate GitHub Organisation. All repositories under github.com/NikiDigitals.
**Rationale:** Single professional identity. Simpler to manage. One account, one portfolio.

### ADR-000016 — Phase 5 Renamed — Mastery, Publication & Optionality
**Decision:** Phase 5 is renamed. Productisation removed as primary objective.
**Rationale:** Career development as Finance Transformation Architect is the primary goal. Scaling is an option, not a target.

### ADR-000018 — Always Choose the Best Long-Term Option
**Decision:** Every decision evaluated against the five-year horizon. Short-term convenience is never the deciding factor.
**Rationale:** LES is a five-year project. Short-term optimisation produces long-term technical debt.

### ADR-000019 — Repository Names Use Full Uppercase Suffix
**Decision:** All repositories follow the convention `LES-[UPPERCASE]`.
**Rationale:** Consistent naming across 18 repositories. Uppercase distinguishable from file and folder names.

### ADR-000021 — One TEMPLATE.md Per Folder
**Decision:** Each folder contains one TEMPLATE.md specific to that folder's document type.
**Rationale:** Self-contained scaffolding. No cross-referencing required at scale.

### ADR-000022 — Footer Three-Column Structure
**Decision:** Website footer: brand left, navigation centre, social icons right.
**Rationale:** Professional web convention. Clean separation of identity, navigation, and social.

### ADR-000023 — GitBook Entries at Defined Trigger Points Only
**Decision:** GitBook content published only at defined trigger points — not after every session.
**Rationale:** Defined triggers ensure every published entry is complete, standalone, and worth a reader's time.

### ADR-000024 — Styling Pass Deferred to Dedicated Session
**Decision:** Website styling applied in one dedicated pass after all content is written.
**Rationale:** Content first, styling second. One-pass styling produces consistent results.

### ADR-000026 — LES-MENTOR as Home Base Chat
**Decision:** LES-MENTOR is the home base for all session reporting and master context updates. One chat per week.
**Rationale:** Prevents context fragmentation across multiple chats.

### ADR-000027 — Claude Chats Are Temporary Workspaces
**Decision:** All outputs committed to GitHub. If it is not in GitHub, it did not happen.
**Rationale:** Claude has no persistent memory. Only committed work is permanent.

### ADR-000028 — Session Journal as Documentation Habit
**Decision:** One journal entry per week covering all sessions that week.
**Rationale:** Weekly consolidation provides a complete picture without the overhead of daily writing.

### ADR-000029 — No Additional Note-Taking Tools
**Decision:** Markdown files in GitHub are sufficient at this stage.
**Rationale:** Tool proliferation adds complexity. LES-KNW will serve this function from Phase 3.

### ADR-000030 — WordPress Website Scope — Content Only
**Decision:** All pages written as markdown before being built in WordPress. Content justifies the page.
**Rationale:** Prevents the failure mode of visually impressive sites with no substance.

### ADR-000032 — GitHub No-Doubles Rule
**Decision:** Reader-facing documents go to GitBook. Working documents stay on GitHub.
**Rationale:** Prevents maintenance overhead of identical content in two places.

### ADR-000035 — Universal Folder Structure Expanded with src/, tests/, specs/
**Decision:** src/ for application code, tests/ within src/, specs/ within docs/ for data models.
**Rationale:** Industry-standard structure. Separates code, tests, and specifications cleanly.

### ADR-000037 — Repository Names Confirmed
