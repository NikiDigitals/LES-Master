## Architecture Decisions

### ADR-000061 — LES-KNW Identified as Future Native Documentation Database
**Decision:** LES-KNW absorbs the documentation function from Phase 3 onwards. GitHub and GitBook serve Phase 0–2. Current documentation structure is deliberately designed to mirror the eventual LES-KNW schema.
**Rationale:** The knowledge management module is the natural permanent home for LES documentation. Designing current structure to mirror the schema makes Phase 3 migration an import, not a restructure. This is a long-horizon architectural decision that requires no immediate action but should inform every documentation structure decision made in Phases 0–2.
**GitBook:** Yes

### ADR-000062 — Diagramming Standards Adopted — C4, BPMN 2.0, ERD, Azure Icons
**Decision:** C4 Model for system architecture diagrams. BPMN 2.0 for process flows. ERD notation for data models. Azure Architecture icons for infrastructure diagrams. draw.io as primary tool for all types.
**Rationale:** No single notation covers all diagram types required by a production-grade ERP. Each standard is adopted for the domain where it is the recognised professional convention. draw.io supports all four natively, is free, and exports to SVG and PNG for documentation embedding. Reference: c4model.com, omg.org/spec/BPMN.
**GitBook:** Yes

## Operational Decisions

### ADR-000063 — ADR File Convention — One File Per Day, Grouped by Subject
**Decision:** ADR files follow the naming convention ADR-YYYY-MM-DD.md. One file per day. All decisions made that day are recorded within the file, grouped into four sections: Architecture Decisions, Strategic Decisions, Technology Decisions, Operational Decisions.
**Rationale:** One file per decision produces an unmanageable number of files at scale. One dated file per day is manageable, traceable, and preserves the chronological record. Grouping by subject within the file makes decisions findable without requiring a separate index lookup.
**GitBook:** No

### ADR-000064 — Journal Entry Cadence — One Entry Per Week
**Decision:** One LES-JOUR journal entry is written per week at the end of the week. It covers all sessions across all days that week. Published to GitBook when the content is substantial, standalone, and reader-facing.
**Rationale:** Weekly consolidation produces more substantial and readable entries than daily fragments. End-of-week timing captures the full picture of what was done and learned. Daily entries for a project of this scope produce low-value repetitive documentation.
**GitBook:** No
